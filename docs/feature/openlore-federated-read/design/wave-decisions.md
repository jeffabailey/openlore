## DESIGN-wave decisions

| # | Decision | Rationale | Status | Locks |
|---|---|---|---|---|
| WD-26 | Slice-03 is a port-and-table EXTENSION of slice-01, not a re-architecture. No new crates; the 8 production crates + xtask from slice-01 are extended in place. | Conservative scope; slice-03 validates a federation thesis ON TOP of the proven walking skeleton. A new crate would require its own ADR per Component Inventory in the cross-feature SSOT and add maintenance surface for marginal value. The crate count cap (~10) noted in ADR-009 holds. | LOCKED | DELIVER MUST extend existing crates; introducing a new crate during DELIVER requires returning to DESIGN. |
| WD-27 | Peer storage uses a NEW port (`PeerStoragePort`), NOT an extension of `StoragePort`. The new port shares the underlying DuckDB connection pool with the existing `DuckDbStorageAdapter` but exposes an orthogonal contract surface. | The slice-03 peer surface (subscribe-state, soft-vs-hard remove, attribution-preserving writes) is genuinely different from author storage (the user is the sole author; no attribution surface). Folding into `StoragePort` would either bloat it or pollute the slice-01 mental model. The two ports are TWO separate Earned Trust contracts; both ship a `probe()`. | LOCKED | DELIVER implements `PeerStoragePort` as a distinct trait per ADR-009 invariant I-4. |
| WD-28 | Peer-PDS read surface is an EXTENSION of `PdsPort` (NOT a new port). The existing `PdsPort` gains `list_peer_records(peer_did, peer_pds_endpoint, cursor)` and `get_peer_record(peer_did, peer_pds_endpoint, rkey)`. | Both methods are XRPC reads against an ATProto PDS; the boundary they cross is the same (ATProto over HTTPS). The asymmetry (own-PDS reads are authenticated; peer reads are unauthenticated) is internal to the implementation; the trait surface is uniform. A new `PeerPdsPort` would split a single concern across two ports with no compositional benefit. | LOCKED | DELIVER extends `PdsPort` in `ports` crate; the `adapter-atproto-pds` impl gains the new methods. Peer PDS endpoint is passed per call (not cached) per ADR-016. |
| WD-29 | Peer DID resolution is an EXTENSION of `IdentityPort` (NOT a new port). The existing `IdentityPort` gains `resolve_peer(did) -> PeerInfo`. | DID resolution is identity-domain work; it already lives in `IdentityPort` (which resolves the user's own DID for signing). Splitting into `IdentityPort` (own) + `PeerIdentityPort` (peer) duplicates infrastructure (PLC client, DID-document parsing) for no boundary benefit. | LOCKED | DELIVER extends `IdentityPort` and `adapter-atproto-did`. |
| WD-30 | Anti-merging invariant (I-FED-1) is enforced at THREE semantically orthogonal layers (type / structural / behavioral), mirroring ADR-009's probe-contract enforcement model. | A single-layer bypass is caught by the other two. The invariant is load-bearing for KPI-FED-1 + KPI-FED-2; one layer of enforcement is insufficient at this stakes level. | LOCKED | DELIVER ships `FederatedRow` with non-Option `author_did` (type); extends `xtask check-arch` with `no_cross_table_join_elides_author` rule (structural); ships integration test `federation_attribution_preserved` (behavioral). |
| WD-31 | `peer_claims` storage is a SINGLE DuckDB file with NEW tables (WD-19 inherited), with a parallel `peer_claims/<peer_did>/<cid>.json` filesystem tree partitioned by peer DID. | The DB layout is per WD-19; the filesystem partition by peer DID makes hard-purge a directory removal (cheaper + easier to audit than per-file dance). Symmetry with slice-01's `claims/<cid>.json` flat layout is broken intentionally because purge semantics differ. | LOCKED | DELIVER implements the partitioned filesystem layout; ADR-014 documents the rationale. |
| WD-32 | The `reason` field on `org.openlore.claim` is added as a TOP-LEVEL OPTIONAL property (NOT nested in `references[]` entries). | Single field; clean serde; one place to validate length; cleanly omitted when None (preserves CID stability for slice-01 claims). Nested placement would multiply complexity for future sugar verbs (`corrects`, `supersedes` would each want their own reason). | LOCKED per ADR-015. | DELIVER implements per ADR-015 §Serde representation. |
| WD-33 | Counter-claim authoring REUSES `VerbClaimPublish` internals via function call (WD-22 inherited). `VerbClaimCounter` constructs the unsigned claim with the right `references[]` + `reason` then dispatches through the slice-01 sign+publish pipeline. | Preserves ADR-003 single-publish-path invariant; any future change to publish behavior applies to counter-claims for free. | LOCKED per ADR-013 + ADR-008. | DELIVER MUST NOT introduce a parallel publish code path; code review + cli probe #5 from component-boundaries enforce this. |
| WD-34 | Self-counter rejection happens in PURE CORE (`claim-domain::validate_counter_claim`), invoked by the CLI verb BEFORE compose preview is rendered. The function takes a `&dyn ClaimLookup` so it can resolve target_cid against EITHER `claims` OR `peer_claims`. | Belt-and-braces: CLI rejects pre-compose for UX (clear error before user sees a useless preview); domain layer rejects at sign-time as a safety net. Same pattern as slice-01's self-reference detection. | LOCKED | DELIVER implements `validate_counter_claim` in `claim-domain` per component-boundaries spec. |
| WD-35 | `--reason` text is NFC-normalized at compose time via `claim-domain::normalize_reason` (a pure function). The normalized bytes are what get signed AND displayed in the preview. | NFC normalization is mandatory for CID determinism (per ADR-006: two visually-identical strings with different normalization forms would produce different CIDs, breaking copy-paste workflows). | LOCKED per ADR-015. | DELIVER adds `unicode-normalization` crate to `claim-domain`; whitelist in `xtask check-arch` pure-core allowlist. |
| WD-36 | The `--purge` confirmation prompt REQUIRES an interactive TTY (WD-21 inherited). `--no-tty` mode REFUSES to run the `--purge` branch and prints an error directing the user to remove `--no-tty` or wait for slice-04's `--yes` flag. | The destructive-confirmation safety valve must hold against accidental scripting at this stage of dogfeed maturity. Auto-confirming `--purge` in `--no-tty` would defeat the J-003c trust promise. | LOCKED | DELIVER implements the refusal branch; integration test asserts `--no-tty --purge` exits non-zero with the directing error message. |
| WD-37 | `peer pull` runs sequentially across peers (per ADR-016). Concurrency is deferred to slice-04. Per-peer pulls fail soft (skipping); per-record pulls fail soft (rejecting); overall exit code is non-zero on ANY skip or rejection. | Sequential keeps per-peer progress output ordered, simplifies failure aggregation, and avoids interleaved error reporting. The performance cost is acceptable for slice-03 peer counts (handful per user × hundreds of claims per peer). | LOCKED per ADR-016. | DELIVER implements sequential pull; slice-04 may revisit with bounded parallelism. |
| WD-38 | Cached peer claims are NOT re-verified at query time in slice-03 (per ADR-016). Re-verification is deferred to slice-04 via a future `peer verify --all` verb. | Pull-time verification + filesystem permissions + the future `peer verify --all` verb form layered defense. Re-verifying every row at every federated query would be correct-er but at significant performance cost. | LOCKED per ADR-016 + accepts the anxiety scenario 2 risk explicitly. | DELIVER implements pull-time-only verification; query path trusts the local store. DISTILL's `# DISTILL: confirm` flag in gherkin scenario "Cached peer claim with corrupted signature is flagged at query time" is RESOLVED: slice-03 does NOT implement query-time re-verification; that scenario is deferred to slice-04. |
| WD-39 | First-pull / first-federated-query / first-counter-claim orientation messages fire ONCE PER USER, gated by RFC3339 timestamps in `~/.config/openlore/identity.toml` under `[federation]` section (per OD-FED-2 accepted default). | Habit-bridging affordance per gherkin-scenarios-expanded.md. Once-per-user (not once-per-session) because the messages are introductions, not reminders. Local-only state; no telemetry implication. | LOCKED per OD-FED-2. | DELIVER implements `cli::OrientationState`; integration test `orientation_fires_once` asserts the three messages each fire on first invocation and not on subsequent. |
| WD-40 | A peer publishing a record whose `author` field is the local user's DID is REJECTED at pull time with `PeerStorageError::SelfAttribution`. Even if such a record's signature DID verify against the current user's key (which would indicate key compromise), the pull-time rejection holds. | Defense against a confused or malicious peer publishing records that claim the current user's authorship. The slice-03 trust model is "we trust signatures, not author fields"; SelfAttribution-rejection is the policy enforcement at the storage write boundary. | LOCKED | DELIVER implements `DuckDbPeerStorageAdapter::write_peer_claim` to reject `author_did == identity.author_did()` with `PeerStorageError::SelfAttribution`. Probe #4 of `PeerStoragePort.probe` asserts this. |
| WD-41 | A peer publishing a record whose `author` field references a DID OTHER than the subscribed peer's DID is REJECTED at pull time (per anxiety scenario 1.2 `# DISTILL: confirm`). The slice-03 trust model is "subscribing to a peer means accepting THEIR claims; cross-attributed records are out of scope for slice-03." | Simpler trust model: the user explicitly chose to follow this peer DID; records from OTHER DIDs on this peer's PDS are treated as "not subscribed to that author." Storing them with attribution to the actual author would create a back-door to "follow Rachel and you automatically also follow whoever Rachel cross-publishes for." Re-evaluate in slice-04 if a JTBD emerges. | LOCKED. RESOLVES DISTILL `# confirm` on anxiety scenario 1.2. | DELIVER implements pull-time check: if `signed.author != subscribed_peer.did`, reject with a `PeerStorageError::CrossAttribution` variant (added to PeerStorageError enum). |
| WD-42 | The inline counter-claim TEMPLATE in `graph query --federated` output (per gherkin scenario "User reads a peer claim they disagree with; counter-claim is one copy-paste away") is shown by default, NOT gated behind `--verbose` (`# DISTILL: confirm` in habit scenario 2). | Habit-bridging is the gherkin scenario's explicit purpose; gating it behind `--verbose` would defeat the point. Output cost is small (~3 lines per peer claim row); P-002 + P-001 personas are CLI-comfortable. | LOCKED. RESOLVES DISTILL `# confirm` on habit scenario 2 inline template. | DELIVER renders the template inline on every peer claim row in `graph query --federated`. May add `--no-counter-tip` flag in slice-04 if dogfeed feedback indicates noise. |
| WD-43 | The one-time first-counter-claim framing block fires EXACTLY ONCE (gated by `federation.first_counter_claim_completed_at` per WD-39), NOT the first 3 times (`# DISTILL: confirm` in habit scenario 2 framing block). | Symmetric with the other two orientation messages; once-per-user is the design pattern. "First 3 times" would create three different rendering paths for `claim counter` (with-framing-1, with-framing-2/3, without) — additional complexity for marginal habit-bridging benefit. | LOCKED. RESOLVES DISTILL `# confirm` on habit scenario 2 framing block trigger. | DELIVER renders the framing block only on the first invocation per install. |
| WD-44 | `peer pull` does NOT auto-notify the target peer of authored counter-claims (anxiety scenario 4 `# DISTILL: confirm`). Notifications are explicitly out of scope per story-map.md deferred-features list. | Slice-03 ships no notification mechanism in either direction; the peer learns about a counter-claim only when they pull from the current user (if they subscribe back). This is the J-003b promise: counter-claims are public structured artifacts, not adversarial pings. | LOCKED. RESOLVES DISTILL `# confirm` on anxiety scenario 4 "no auto-notification." | DELIVER ships NO notification code path; integration test asserts publishing a counter-claim emits no network traffic to the peer's PDS beyond the user's normal own-PDS publish. |
| WD-45 | The four DESIGN-wave ADRs (013-016) are accepted with this DESIGN-wave handoff; no further DESIGN iterations required pending peer review. | Each ADR has 2+ alternatives considered, locked alternatives rationale from DISCUSS, and Earned Trust sections that translate to concrete probe contracts. The slice-03 architecture is a straightforward extension of slice-01; no novel architectural risk. | LOCKED pending Atlas (solution-architect-reviewer) approval. | Reviewer may flag issues for an iteration-2 pass. |

## DISTILL `# confirm` resolutions (consolidated)

The DISCUSS-wave gherkin-scenarios-expanded.md flagged several behaviors
for DESIGN resolution. The DESIGN decisions above resolve each:

| `# DISTILL: confirm` flag | Location in gherkin-scenarios-expanded.md | Resolved by |
|---|---|---|
| Cross-attributed records: store vs reject | Anxiety scenario 1.2 | **WD-41 LOCKED**: reject at pull time with `PeerStorageError::CrossAttribution` |
| Query-time re-verification of cached peer claim signatures | Anxiety scenario 2 | **WD-38 LOCKED**: NOT in slice-03; deferred to slice-04 `peer verify --all` verb |
| `peer audit <did>` verb | Anxiety scenario 3 | **OD-FED-3 LOCKED DISCUSS-side**: defer to slice-04; derivable from `peer list --include-purged` + `graph query --include-counters` (both deferred) |
| No auto-notification of retraction to peer | Anxiety scenario 4 | **WD-44 LOCKED**: no notification mechanism in slice-03 in any direction |
| First-pull orientation trigger | Habit scenario 1 | **WD-39 LOCKED**: once-per-user via `identity.toml` state |
| Inline counter-claim template visibility | Habit scenario 2 | **WD-42 LOCKED**: shown by default; no `--verbose` gate |
| First-counter-claim framing block trigger | Habit scenario 2 | **WD-43 LOCKED**: once-per-user (not first-3-times) |

DISTILL inherits these resolutions as binding; acceptance tests assert
the locked behavior, not the open alternatives.
